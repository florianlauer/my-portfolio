# Codebase Concerns

**Analysis Date:** 2026-03-07

## Tech Debt

**Duplicated Constants Between Gallery Components:**

- Issue: `INITIAL_COUNT` and `LOAD_MORE_COUNT` are defined in both `src/components/gallery/GalleryClient.tsx` (lines 9-10) and `src/components/gallery/GalleryGrid.tsx` (lines 7-8). These must stay in sync manually.
- Files: `src/components/gallery/GalleryClient.tsx`, `src/components/gallery/GalleryGrid.tsx`
- Impact: If one is changed without the other, the staggered animation delay logic in `GalleryGrid` breaks (it uses `INITIAL_COUNT` to detect new batches at line 97).
- Fix approach: Export both constants from `GalleryClient.tsx` (or a shared `gallery/constants.ts`) and import them in `GalleryGrid.tsx`.

**Untyped Third-Party Module Declaration:**

- Issue: `react-responsive-masonry` has a bare `declare module` with no type information at all. All props are effectively `any`.
- Files: `src/types/react-responsive-masonry.d.ts`
- Impact: No type safety when using `<Masonry>` or `<ResponsiveMasonry>` components. Typos or wrong props are not caught at compile time.
- Fix approach: Write a proper type declaration with `ResponsiveMasonryProps` and `MasonryProps` interfaces covering the props actually used (`columnsCountBreakPoints`, `gutterBreakPoints`, `children`).

**Duplicated Page Shell Logic in Home Page:**

- Issue: `src/app/page.tsx` manually assembles the same shell (GlobalBackground, skip-to-content link, HomeNav, container layout) that `src/components/page-shell/PageShell.tsx` provides for secondary pages. The skip-link target differs (`#content-start` in home vs `#contenu` in PageShell).
- Files: `src/app/page.tsx`, `src/components/page-shell/PageShell.tsx`
- Impact: Changes to shared layout (e.g., z-index, padding, skip-link behavior) must be applied in two places. The inconsistent skip-link anchors may confuse accessibility testing.
- Fix approach: Refactor `PageShell` to accept home-page variations (e.g., no wrapping card) or extract the shared boilerplate into a `LayoutShell` wrapper. Standardize the skip-link target.

**Gallery Content as Giant Static Array:**

- Issue: `src/content/gallery.ts` is a 513-line file with 85 manually maintained image entries. The `alt` and `caption` fields are identical for every entry. Adding images requires both copying a file to `public/gallery/` and updating this array.
- Files: `src/content/gallery.ts`
- Impact: Tedious to maintain, easy to forget one step. Redundant alt/caption doubles the per-entry boilerplate.
- Fix approach: Consider generating the gallery list from the filesystem at build time (e.g., a script that reads `public/gallery/`, extracts metadata from filename conventions). At minimum, derive `alt` from `caption` automatically in the type/component.

**Dark Mode Defined But Not Toggleable:**

- Issue: A complete `.dark` theme is defined in `src/app/globals.css` (line 98), but there is no dark mode toggle, no `next-themes` integration, and no `class` strategy on `<html>`. The dark palette is unreachable.
- Files: `src/app/globals.css`, `src/app/layout.tsx`
- Impact: Dead CSS code. If dark mode is added later without testing, the palette may have contrast issues since it has never been visually validated.
- Fix approach: Either remove the `.dark` block until dark mode is actually implemented, or add `next-themes` with a toggle component. If keeping it, add a `@custom-variant dark` that uses `prefers-color-scheme` as a fallback.

**Sidebar Theme Variables Never Used:**

- Issue: The CSS defines a full set of `--sidebar-*` custom properties (sidebar, sidebar-foreground, sidebar-primary, sidebar-accent, sidebar-border, sidebar-ring) in both light and dark themes. There is no sidebar component in the codebase.
- Files: `src/app/globals.css` (lines 88-95, 123-130)
- Impact: Unnecessary CSS bloat (12 unused custom properties). Likely generated by shadcn/ui scaffolding.
- Fix approach: Remove all `--sidebar-*` variables and their `--color-sidebar-*` theme mappings. Also remove `--chart-*` variables if no chart components are planned.

## Known Bugs

**Lightbox Index Shows Wrong Number After Shuffle:**

- Symptoms: The lightbox shows `idx + 1 / n` (line 177 in `Lightbox.tsx`) where `idx` is the position in the 3-slide virtual array (`prevIndex`, `currentIndex`, `nextIndex`), not the user-visible position. For prev/next slides the counter shows the wrong number.
- Files: `src/components/gallery/Lightbox.tsx:177`
- Trigger: Open lightbox, look at the caption on the previous/next slide visible during swipe.
- Workaround: Only the center slide is fully visible normally, so most users do not notice.

## Security Considerations

**Email Address Exposed in Source:**

- Risk: `contact@flauercase.dev` is hardcoded in `src/content/site.ts` and rendered as a `mailto:` link. This makes it trivially scrapable by bots.
- Files: `src/content/site.ts:15`, `src/components/home-sections/ContactSection.tsx:49`
- Current mitigation: None.
- Recommendations: Accept the risk (common for portfolio sites) or obfuscate the email (e.g., render via JS, use a contact form, or encode with character entities).

**External Image Hosts Allowed:**

- Risk: `next.config.ts` allows remote images from `upload.wikimedia.org`. This is used for flag SVGs in `JourneySection`. If Wikimedia URLs change or serve unexpected content, it could affect rendering.
- Files: `next.config.ts`, `src/components/home-sections/JourneySection.tsx:23-33`
- Current mitigation: Only two specific flag images are loaded.
- Recommendations: Download the flag images to `public/` for full control and offline resilience, eliminating the remote pattern allowance.

**No Content Security Policy Headers:**

- Risk: No CSP headers are configured. The site uses `dangerouslySetInnerHTML` for JSON-LD (which is safe for structured data but still a pattern to watch).
- Files: `src/app/layout.tsx:87`
- Current mitigation: No user-generated content exists, so XSS risk is minimal.
- Recommendations: Add basic CSP headers via `next.config.ts` headers or middleware when deploying to production.

## Performance Bottlenecks

**Large Unoptimized Gallery Images Served via Native `<img>`:**

- Problem: The gallery grid uses native `<img>` tags (not Next.js `<Image>`) for masonry layout. 85 images totaling 73 MB are served from `public/gallery/`. Some "compressed" images are still 2+ MB (e.g., `IMG_7622.compressed.jpeg` at 2.6 MB). The first 20 are loaded eagerly.
- Files: `src/components/gallery/GalleryGrid.tsx:115`, `public/gallery/`
- Cause: `react-responsive-masonry` requires native `<img>` for proper layout calculation. Images are pre-compressed but at their original resolution, with no responsive `srcset` or WebP conversion.
- Improvement path: Generate multiple sizes and WebP variants at build time (e.g., with `sharp` or Next.js image optimization API). Add `srcset` and `sizes` attributes to the native `<img>` tags. Target max 200-400 KB per thumbnail. The lightbox already uses `next/image` with optimization, which is good.

**Three Full-Screen Background Images Loaded on Every Page:**

- Problem: `GlobalBackground` loads 3 large slideshow images on every page (home, a-propos, galerie). `asset-rio.jpeg` is 1.1 MB. Only the first image is visible initially; the other two wait 10s/20s to appear.
- Files: `src/components/global-background/GlobalBackground.tsx:7-11`, `public/asset-*.jpeg`
- Cause: All three `<Image>` elements are rendered on mount. Only the first has `priority`.
- Improvement path: Lazy-load the second and third images. Preload them after a delay or use `loading="lazy"` on non-active slides. Consider converting to WebP/AVIF.

**useScrollY Hook Triggers Re-renders on Every Scroll Frame:**

- Problem: `useScrollY` calls `setScrollY(window.scrollY)` on every animation frame during scroll, causing the consuming component to re-render each frame.
- Files: `src/hooks/useScrollY.ts`
- Cause: React state update on every rAF callback. Currently only used by the unused hook (no direct consumer found in the codebase besides the hook definition).
- Improvement path: If this hook is not actively used, remove it. If used in the future, prefer a ref-based approach that updates CSS custom properties directly (like `GlobalBackground` and `HomeNav` already do with `el.style.transform`).

**Gallery Shuffle on Client Mount:**

- Problem: `GalleryClient` shows a loading placeholder during SSR/hydration, then shuffles and re-renders the entire gallery after mount. This means the gallery is never part of the initial HTML, hurting First Contentful Paint and SEO for gallery images.
- Files: `src/components/gallery/GalleryClient.tsx:33-38`, `src/app/galerie/page.tsx:5-8`
- Cause: `react-responsive-masonry` is client-only (uses `window` for breakpoints). The shuffle also requires client-side JS.
- Improvement path: Server-render the first batch in a simple CSS grid as a fallback, then hydrate into masonry. Or use a seed-based shuffle that can run on the server.

## Fragile Areas

**HomeNav Scroll-Driven Animation:**

- Files: `src/components/home-nav/HomeNav.tsx`
- Why fragile: The component manually manages 5 CSS custom properties via `style.setProperty()` in a scroll handler (lines 107-111), bounce state via `queueMicrotask` + `setTimeout`, and section intersection via 4 independent `IntersectionObserver` instances. All values are magic numbers (`SCROLL_END_PX = 280`, `AT_MIN_THRESHOLD = 0.98`, etc.).
- Safe modification: When changing nav sizing, update the magic numbers in the constants at the top of the file. Test on mobile (uses different `SCROLL_END_PX_MOBILE = 140`). The bounce animation CSS is in `src/app/globals.css` (lines 181-223) and must stay in sync with `BOUNCE_DURATION_MS`.
- Test coverage: None. Manual testing required across screen sizes.

**Lightbox Swipe + Transition Coordination:**

- Files: `src/hooks/useSwipe.ts`, `src/components/gallery/Lightbox.tsx`
- Why fragile: The swipe mechanism relies on a `transitionEnd` event to trigger the actual slide change (`handleTransitionEnd` at line 101 in `useSwipe.ts`). If the CSS transition is interrupted or does not fire (e.g., `display:none`, rapid swipes), the navigation callback is lost and the lightbox gets stuck.
- Safe modification: Always ensure the animated element stays in the DOM and visible during transitions. Do not add `will-change: transform` or other properties that might affect transition event firing.
- Test coverage: None.

**Gallery Image Load Counting:**

- Files: `src/components/gallery/GalleryGrid.tsx:48-53`
- Why fragile: The loading state relies on counting `onLoad` events from all images. If any image fails to load (404, network error), `handleImageLoad` is never called for it, and `loadedCountRef` never reaches `expectedCountRef`. The 2.5s fallback timeout (line 44) is the only safety net.
- Safe modification: Add `onError` handler that also increments the counter.
- Test coverage: None.

## Scaling Limits

**Static Gallery Content:**

- Current capacity: 85 images, 513 lines of TypeScript.
- Limit: At 200+ images, `src/content/gallery.ts` becomes unwieldy. The entire array is imported into the server component and passed as props to the client. For very large galleries, this serialized data adds to the page bundle.
- Scaling path: Generate gallery data from filesystem at build time. Consider pagination with server-side data fetching instead of passing the full array as props.

**Public Directory Size:**

- Current capacity: 75 MB total in `public/`.
- Limit: At 200+ full-resolution images, the public directory exceeds typical Vercel deployment limits (output size). Git repository also becomes unwieldy.
- Scaling path: Move images to a CDN or object storage (e.g., Cloudflare R2, S3). Use `next/image` with a remote loader.

## Dependencies at Risk

**react-responsive-masonry:**

- Risk: No TypeScript types. The bare `declare module` means zero type safety. The package forces use of native `<img>` instead of `next/image`, losing Next.js image optimization in the grid view.
- Impact: Gallery grid has no type checking. Images are served unoptimized.
- Migration plan: Consider CSS-only masonry (CSS `columns` or `masonry` layout when browser support improves) or a typed alternative like `react-masonry-css`. Alternatively, implement a custom masonry layout with CSS Grid.

**prop-types:**

- Risk: Listed as a production dependency but never imported anywhere in the source code. Likely a transitive dependency that was accidentally promoted to direct.
- Impact: Unnecessary bundle size (minor).
- Migration plan: Remove from `dependencies` in `package.json`. If a transitive dependency needs it, it will be resolved from their own `node_modules`.

## Missing Critical Features

**No Error Boundaries:**

- Problem: No React error boundaries exist. If a client component throws (e.g., gallery fails to load), the entire page crashes with a white screen.
- Files: All client components in `src/components/`
- Blocks: Graceful degradation on component errors.

**No Analytics or Performance Monitoring:**

- Problem: No analytics (Vercel Analytics, Plausible, etc.) and no error tracking (Sentry, etc.) are configured. Production issues would go undetected.
- Files: `src/app/layout.tsx`
- Blocks: Understanding visitor behavior and catching runtime errors.

**No 404 Page:**

- Problem: No custom `not-found.tsx` exists. Next.js will show its default 404 page, which does not match the site design.
- Files: `src/app/` (missing `not-found.tsx`)
- Blocks: Consistent user experience on invalid routes.

## Test Coverage Gaps

**No Tests Exist:**

- What's not tested: The entire codebase. No test runner is configured (no Jest, Vitest, or Playwright).
- Files: All files in `src/`
- Risk: Any refactoring of gallery logic, scroll animations, or content changes could introduce regressions unnoticed. The lightbox swipe logic and image load counting in particular have subtle edge cases that would benefit from unit tests.
- Priority: Medium. For a static portfolio site the risk is manageable, but the gallery components (`GalleryClient`, `GalleryGrid`, `Lightbox`, `useSwipe`) contain enough logic to warrant at least unit tests for the state management.
